Regan Koopmans

Dependencies and Abstractions

Commercial software increasingly involves many layers of complicated dependency management and abstractions. I would like to address why I think this is a problem:

Dependencies

In my view, it is an anti-pattern that large established software companies depend on hobby software written by a small group of contributors, if not a single individual. I think that collaboration in software is a great idea, but we should be more realistic and accept the state of most open-source libraries. Here I am not talking about established open-source projects backed by companies, non-profits and foundations. I am speaking specifically about libraries that would be very easy to in-house, but nevertheless are externalised to the pet projects of people in their spare-time.

Companies use open-source code because it saves time and money. Code borrowed is code that does not need to be written. It is too tempting for most businesses to resist. You download 100 dependencies using npm, write some glue code, and your software solution is near complete. However, over time the decision to outsouce this work inhibits your own code. You begin to spend more time solving dependency issues than the problems that are core to your business. Every new dependency you add exponentially increases the number of possible conflicts and incompatibilities. Your dependencies have their own dependencies, and at some point the combinatorial problem begins to exceed the ability for both humans and tooling to solve. Most engineers would far rather spend time implementing some functionality than live in dependency hell. The bespoke solution would naturally fit the requirements of your code better.

Dependencies also exist outside your control. You do not know whether they will exist in the next 5, 10, 20 years. I am extremely skeptical that a Python project made today, without any active work, will still run in 10 years. If we are not writing software to last at least that long, why not just write your software in Excel? You should "own" as much of the code as possible. The problem of dependencies is confounded by the next topic: Abstractions.

Abstractions

Much software written today exists on a teetering tower of abstractions. It's considered perfectly reasonable to have a Java program, running on a JVM, running on an OS, running on a Docker container, running on an OS, running on a virtual machine, running in a data-center somewhere. Excel is more "native" than a lot of commercial software, even if you use the online version. There are multiple layers in which dependency hell can occur, most prominently at the application and the first OS level. You can have incompatibilities across these layers too.

All of this is fine if you stick to extremely "vanilla" setups that cloud providers have outlined. Your application can be deployed and geographically replicated in minutes. But this setup can never be considered "stable". We live in a strange world where C code from the 80s can be compiled to modern machines, but a JavaScript project from 2010 is not very likely to run.

I am not really sure where I am going with this argument. I suppose I just want more simplicity.
